    Synthèse exécutive et décisions d’architecture
    Ce projet vise à créer un service local en C# (.NET 8 / ASP.NET Core) exposé sur http://127.0.0.1:12345 avec :
    REST (lecture/commande) + WebSocket (événements temps réel) + Swagger UI (doc + test). [1]
    Une capacité “standalone” (chaque borne fonctionne seule), et un mode ArcadeHub optionnel (centralisateur LAN) avec rôle Hub/Master et Node/Client (opt‑in). [2]
    Une ambition claire : devenir la source canonique (state + média + règles) utilisée par des clients spécialisés (marquee, DMD, LED panel, molding RGB, scoreboard, multi‑écrans), en migrant progressivement tes services actuels (Marquee Manager / LedPanel Manager) vers des clients API.
    Le point structurant issu de tes systèmes actuels et des inspirations étudiées : l’API ne doit pas seulement “exposer de la donnée”, elle doit exposer un état canonique + un moteur d’intentions (scènes) + un moteur de règles réactives, afin de supporter des comportements avancés déjà présents dans tes formats :
    LIP (ton DSL) : événements d’inputs physiques (press/release) déclenchant des macros (set_panel_colors, restore_panel, set_button…), donc un rule engine. (exemples fournis par toi)
    LAY (layouts marquee réactifs aux outputs MAME) : binding d’éléments graphiques à des outputs (ex : genout5) pour allumer/éteindre des overlays de lampes, donc un rule engine aussi, mais côté affichage. (exemple fourni par toi)
    Outputs MAME : flux key=value via -output network (port 8000 fixé côté MAME) alimentant des effets temps réel. [3]
    Événements EmulationStation : scripts/événements (game-start, game-end, game-selected…) déjà formalisés dans ton events.ini (host/port + liste d’actions). [4]
    Enfin, tu demandes explicitement (a) push d’une vue (jeu/système/collection) d’un poste vers une borne (marquee + layout + éventuellement launch), (b) centralisation des dossiers XML/LIP, pools de scrap, command.dat et autres fichiers clés dans l’API, (c) réflexion “score courant” (idéalement) et scoreboard global, et (d) pilotage LED complet (boutons, ring, molding RGB…) en s’inspirant de Pixelcade sans l’intégrer.
    Note de transparence : tu as fourni api.zip (début d’API). Je n’ai pas pu l’exploiter via l’outil de recherche de fichiers du contexte (pas de résultats indexés). La proposition ci‑dessous est donc consolidée à partir des dépôts et sources web ; elle est compatible avec une intégration/migration de ton brouillon existant, mais sans analyse de son code.

    Leçons de conception
    1) Séparer “State canonique” et “Intent”MarquesasServer expose un état (SelectedGame) ET des assets (images). [11]Ton besoin de “pousser une vue” implique de pouvoir publier une intention qui override l’état naturel (sélection ES) pendant un temps (contest, attract mode, multi‑écrans).
    2) Le temps réel doit être event-driven- Côté ES : scripts/événements. Ton events.ini catalogue des événements riches. [12]- Côté MAME : outputs réseau (-output network) et parseur key=value. [3]
    3) Le “rule engine” est centralLes LIP et les LAY expriment déjà une logique “si trigger alors macro”. L’API doit donc fournir : - un modèle canonique de triggers/actions,- un chargement/validation/versionning des règles,- et un bus temps réel vers les clients qui exécutent (LED / overlay / marquee).
    
    
    
    Inventaire des sources de données locales et ingestion
    
    Sources à supporter en priorité (et cohérence avec tes projets)
    Événements ES / RetroBat - RetroBat peut exécuter des scripts dans des dossiers d’événements (game-start, game-end, shutdown, etc.) situés sous C:\RetroBat\emulationstation\.emulationstation\scripts. [13]- Ton système Marquee Manager formalise events.ini : host, port, et une liste de commandes (game-selected, system-selected, screensaver…) utilisables comme “contrat d’événements”. [4]- LedPanel Manager observe ESEvent.arg (event=game-selected) et en extrait param1 (émulateur) + param2 (rom path). [14]
    Métadonnées jeux - gamelist.xml (EmulationStation) est un pivot : tags standards (name, desc, image, releasedate, players, playcount, lastplayed) et chemins relatifs/absolus selon conventions. [15]- LPEvents.py démontre une stratégie robuste : cache des gamelists et index (system, name|basename) → (emulator, core). [16]
    MAME référentiel + outputs - mame -listxml : export XML exhaustif exploité par frontends/outils. [17]- mame -output network : outputs key=value, port réseau fixe 8000. [7]- Ton listener MAME se connecte à 127.0.0.1:8000, lit des fragments, split \r, déduplique les séquences et transmet. [18]
    Hi-scores - hi2txt décode des hi-scores à partir des fichiers nvram (natif) ou .hi (plugin hiscore + hiscore.dat). [19]- hi2txt-xml existe précisément pour permettre à des logiciels tiers de parser/normaliser des scores pour leaderboard/achievements. [20]- LEDBlinky illustre l’intégration “prête à l’emploi” : il utilise Hi2txt pour décoder les fichiers MAME et peut afficher top1/top3/all, y compris lors d’événements (FE Scroll). [8]
    RetroAchievements - La Web API nécessite une Web API Key et se comporte “comme un mot de passe”. [21]- Endpoint “Get Game Hashes” : mapping par MD5. [22]- Ton script ESRetroAchievements active les logs RetroArch (log_to_file, log_verbosity) et appelle l’API RA via paramètres z/y. [23]
    
    
    Méthodes d’ingestion recommandées
    HTTP ingest (fortement recommandé) : scripts ES → POST /ingest/es (évite les courses/latences des fichiers d’état). Les docs Recalbox rappellent qu’un statefile peut déjà avoir changé si des événements sont rapprochés, et recommandent de vérifier l’action associée. [24]
    FileSystemWatcher : utile en fallback et pour les dossiers hi/nvram/media/caches (déclenchements sur création/modif). [25]
    TCP output provider : pour MAME -output network + futurs output servers (pattern OutputBlaster + UDP discovery). [26]
    
    
    
    Modèle canonique “State + Intent + Rules” incluant LIP et LAY
    
    Concept clé : State (observé) vs Intent (voulu)
    State : ce que la borne vit réellement (sélection ES, jeu en cours, outputs MAME, hi-scores persistés, etc.).
    Intent : ce que l’on désire afficher/activer (ex : “affiche la fiche de ce jeu” même si l’utilisateur browse autre chose ; “mode contest” ; “attract”).Le “winner” entre State et Intent se décide par un arbitre (orchestrator) : priorité, TTL, source (local/hub), règles de sécurité.

    Moteur de règles réactives : un IR canonique commun à LIP et LAY
    Tu as deux DSLs puissants : - LIP : triggers d’inputs physiques (press/release) + macros (set_panel_colors, restore_panel, set_button…).- LAY : binding d’éléments (images) à des outputs MAME (ex : genout5 → afficher lamp overlay).
    La proposition : l’API doit définir un IR (Intermediate Representation) commun :
    
    Triggers canoniques
    input.button.press / input.button.release (ex : B6, B8…)
    mame.output.changed (key genout5, genout6…) [3]
    ui.game.selected / ui.game.started / ui.game.ended (depuis scripts ES/events.ini) [28]
    hiscore.updated (filewatch hi/nvram + hi2txt) [29]
    ra.achievement.unlocked / ra.leaderboard (mode avancé) [30]
    
    
    Actions canoniques (exécutions côté clients)
    led.setColors(panelRef, colors…)
    led.restore(panelRef)
    led.setButton(panelRef, button, color)
    display.setScene(target, sceneSpec) (marquee/dmd/lcd…)
    display.setElementState(sceneId, elementId, state) (cas LAY)
    system.launch(gameRef, options) (commande contrôlée)
    http.get(url) (optionnel, inspiration LEDBlinky “web request GET” via outputs) [8]
    
    
    Exemple IR (inspiré de tes LIP)
    {  &quot;ruleSetId&quot;: &quot;lip:Nintendo64:Arcade-Shark-8B&quot;,  &quot;scope&quot;: { &quot;systemId&quot;: &quot;n64&quot;, &quot;layout&quot;: &quot;8-Button&quot; },  &quot;rules&quot;: [    {      &quot;when&quot;: { &quot;type&quot;: &quot;input.button.press&quot;, &quot;button&quot;: &quot;B8&quot; },      &quot;then&quot;: [        {          &quot;type&quot;: &quot;led.setColors&quot;,          &quot;panel&quot;: &quot;CURRENT&quot;,          &quot;colors&quot;: {            &quot;B1&quot;: &quot;YELLOW&quot;, &quot;B2&quot;: &quot;YELLOW&quot;, &quot;B3&quot;: &quot;YELLOW&quot;, &quot;B4&quot;: &quot;YELLOW&quot;,            &quot;B5&quot;: &quot;GRAY&quot;, &quot;B6&quot;: &quot;GRAY&quot;, &quot;B9&quot;: &quot;GRAY&quot;, &quot;B10&quot;: &quot;RED&quot;          }        }      ]    },    {      &quot;when&quot;: { &quot;type&quot;: &quot;input.button.release&quot;, &quot;button&quot;: &quot;B8&quot; },      &quot;then&quot;: [ { &quot;type&quot;: &quot;led.restore&quot;, &quot;panel&quot;: &quot;CURRENT&quot; } ]    }  ]}
    
    
    Exemple IR (inspiré de ton LAY “chasehq”)
    {  &quot;ruleSetId&quot;: &quot;lay:chasehq:Marquee_Only&quot;,  &quot;scope&quot;: { &quot;systemId&quot;: &quot;mame&quot;, &quot;machine&quot;: &quot;chasehq&quot; },  &quot;assets&quot;: {    &quot;marquee&quot;: &quot;chasehq_marquee.png&quot;,    &quot;redlamp&quot;: &quot;redlamp.png&quot;,    &quot;bluelamp&quot;: &quot;bluelamp.png&quot;  },  &quot;bindings&quot;: [    { &quot;outputKey&quot;: &quot;genout5&quot;, &quot;element&quot;: &quot;redlamp&quot;, &quot;states&quot;: { &quot;0&quot;: &quot;hidden&quot;, &quot;1&quot;: &quot;visible&quot; } },    { &quot;outputKey&quot;: &quot;genout6&quot;, &quot;element&quot;: &quot;bluelamp&quot;, &quot;states&quot;: { &quot;0&quot;: &quot;hidden&quot;, &quot;1&quot;: &quot;visible&quot; } }  ],  &quot;views&quot;: [    { &quot;name&quot;: &quot;Marquee_Only&quot;, &quot;target&quot;: &quot;marquee&quot;, &quot;resolution&quot;: &quot;1920x360&quot; }  ]}
    Ce design répond directement à ton besoin : les LIPs ne sont pas des “configs”, ce sont des règles ; les LAYs aussi. L’API doit donc : - stocker/versionner ces règles, - choisir le bon ruleset selon le contexte (jeu, système, layout physique), - et publier des “intentions d’exécution” aux clients.

    Stratégies RetroAchievements et hi‑scores (avec limites “score courant”)
    
    RetroAchievements : modes passif vs avancé
    Mode passif (MVP)- Pas de clé utilisateur obligatoire. - Observation locale : logs RetroArch (à activer) + logs émulateurs standalones (Aynshe mentionne es_launch_stdout.log et logs PCSX2/DuckStation/Dolphin/PPSSPP). [40]- Objectif : “RA présent/pas présent”, hardcore flag, jeu RA probable.
    Mode avancé (V1)- Stockage de la Web API Key RA et identifiant. RA recommande de traiter la clé comme un mot de passe. [21]- Chiffrement sous Windows via DPAPI (ProtectedData) côté node pour sécuriser les secrets. [41]- Mapping ROM → RA Game : utiliser endpoint de hashes (MD5). [22]- Cache local + throttling (éviter trop d’appels, surtout en mode hub).
    
    
    Hi‑scores : intégration hi2txt et normalisation
    Ce que hi2txt sait faire (fiable aujourd’hui)- Décoder des hi-scores persistés depuis nvram ou .hi + hiscore.dat, et expliquer comment localiser ces fichiers. [42]- Le projet hi2txt-xml est explicitement pensé pour permettre à des logiciels tiers de produire des leaderboards et d’exploiter les scores. [20]- LEDBlinky matérialise un usage proche du besoin : top score/initiales au démarrage, et tokens ~highscore~, ~highscoretop3~, ~highscoreall~. [8]
    Refresh strategies recommandées - Déclenchement principal : sur ui.game.ended (fin de partie) + watchdog sur hi/ &amp; nvram/ (FileSystemWatcher + debounce). [43]- En mode hub/contest : endpoint hub “refresh all nodes” + événement WS hiscore.updated pour recalcul scoreboard.
    “Score courant” pendant la partie : état des lieux - Les outputs MAME (-output network) concernent les “output notifiers” (lampes, LED start, etc.), pas une lecture générique du score mémoire. [7]- hi2txt travaille sur des fichiers persistés (nvram/.hi) et pas sur le score RAM “live”. [42]
    ➡️ Proposition réaliste : - MVP/V1 : scoreboard basé sur hi2txt (persisté, robuste, compatible contests).- V2 : module optionnel “live score” par émulateur via plugins/Lua/memory watch (fortement variable selon jeu/émulateur). Le cahier des charges doit explicitement considérer ce point comme R&amp;D.

    Centralisation média : “API comme source unique”
    Tu veux centraliser : - dossiers XML/LIP (panels),- dossiers de scrap (marquees arcs, ScreenScraper/…), - fichiers clés (command.dat &amp; autres),- et produire des endpoints uniformes (Media API) pour tous les services.
    Le cahier des charges doit donc intégrer un MediaStore avec : - import (copie + hash + index), - versionning (par jeu/système/layout), - conventions de résolution (fallback : game → system → default), - API de consultation (list, get, checksum), - et événements (media.updated).
    Ton READMESETUP décrit déjà des mécanismes de scrap/pool (scrap.pool, scrapfailed.pool) et des patterns d’assets ; la migration naturelle est de déplacer ces pools dans le MediaStore et d’exposer leur état via API. [46]
    
    
    Mode multi‑bornes : standalone, node, hub, master/client
    Objectif : une borne doit rester autonome si le hub est absent.
    Proposition de rôles : - standalone (défaut) : aucune dépendance réseau, tout local. - node : participe à un hub (opt‑in), publie son state, accepte des intents distants selon politique. - hub : agrège états, publie scoreboard global, orchestre contests. - master (optionnel) : un node peut être “master local” pour un îlot (ex : 2 écrans/2 devices), même sans hub.
    Politique de priorité Intent (à définir explicitement) - Local (inputs physiques + navigation utilisateur) priorité haute, sauf mode contest. - Hub peut envoyer un Intent “contest” avec TTL ; l’arbitre choisit IntentHub si priority &gt; local ET scope autorisé.
    Discovery - Liste statique (config) en MVP. - V2 : discovery UDP inspirée OutputBlaster (broadcast sur LAN). [2]
    
    
    Sécurité
    Bind 127.0.0.1 par défaut ; passage LAN explicite (config).
    Auth simple en LAN : token API + scopes (read, control, admin, hub).
    Secrets (RA key, tokens hub) chiffrés DPAPI sur Windows (ProtectedData). [41]

    Discovery LAN (UDP/mDNS) + robustesse réseau. [2]
    Refactor complet des clients (nouveaux Marquee/LedPanel clients API).
    
    
    Risques et limitations
    Variabilité émulateurs : logs/outputs/hiscore diffèrent ; il faut une approche provider‑based et tolérante. [47]
    Hi-scores : tous les jeux ne supportent pas le saving ; hi2txt dépend de hiscore.dat / plugin / nvram. [48]
    Live score : non garanti, à cadrer en V2 R&amp;D (sinon promesse intenable).
    BIOS : contraintes légales/licences ; l’API doit seulement diagnostiquer (hash/nom/emplacement) en s’appuyant sur docs/dat (No‑Intro/Redump/Libretro). [49]
    
    
    
    Recommandations concrètes pour démarrer le développement C
    
    Endpoints prioritaires à implémenter en premier
    1) /api/v1/health, /api/v1/version2) /api/v1/ingest/es + scripts ES “game-selected/game-start/game-end” (compat events.ini) [33]3) WS /ws + publication ui.*4) Provider MameOutputs (TCP 8000) + mame.output.changed [3]5) MediaStore “GET /media/*” + index simple6) Hi2txt provider + GET /hiscores/{gameId} (persisté) [19]
    
    
    Interfaces C# clés (contrats)
    IProvider : Start/Stop + Health + PushEvents
    IEventBus : publish/subscribe typés (Channel ou MediatR)
    IRuleCompiler : compile LIP/LAY → IR
    IRuleEngine : apply(trigger, context) → actions
    IIntentStore : push/pop intents (TTL/prio/source)
    IMediaStore : resolve(GameRef, AssetKind) → MediaRef
    INodeClient / IHubClient : échanges node↔hub
    
    
    Swagger UI
    Swashbuckle fournit l’intégration Swagger UI pour tester routes et schémas directement depuis l’application. [31]
    
    Ce cahier des charges intègre explicitement la puissance de tes LIP (événements/macro) et de tes LAY (marquee réactif aux outputs), en les requalifiant comme un moteur de règles canonique au cœur de l’API. Il consolide aussi l’approche “state + assets” (MarquesasServer), “remote control” (Playnite Web), “outputs/action dispatch” (LEDBlinky, MAME outputs), et “multi‑borne opt‑in” (OutputBlaster discovery pattern), tout en garantissant que chaque borne reste standalone par défaut.
